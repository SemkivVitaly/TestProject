# Полная инструкция: подключение и получение информации через MAVLink с Mission Planner

Подробное руководство по настройке WiFi-моста на ESP32-S3-Box для работы с Mission Planner по протоколу MAVLink: от подключения проводов до отображения телеметрии и параметров.

---

## 1. Общая схема

```
┌─────────────────┐     Wi‑Fi (TCP/UDP)      ┌──────────────────┐     UART      ┌─────────────┐
│ Mission Planner  │  ◄─────────────────────►  │  ESP32-S3-Box    │  ◄──────────►  │  Автопилот  │
│ (ноутбук/ПК)     │   порты 8880 / 14550     │  (WiFi-мост)     │  пины 13/14   │ (Pixhawk…)  │
└─────────────────┘                           └──────────────────┘               └─────────────┘
```

- **Mission Planner** общается с платой по Wi‑Fi (TCP или UDP).
- **ESP32-S3-Box** пересылает байты между Wi‑Fi и UART без изменения.
- По UART плата подключена к **автопилоту** (Pixhawk, Cube и т.п.).
- Протокол обмена — **MAVLink**: телеметрия, параметры, команды идут пакетами.

Чтобы всё работало, нужно: правильно соединить провода, совпадающая скорость UART, один и тот же тип подключения в Mission Planner (TCP или UDP) и корректные настройки в `config.h`.

## 2. Подключение автопилота к ESP32-S3-Box (UART)

### 2.1 Какие пины использовать

| Сигнал | Пин ESP32-S3-Box | Подключение |
|--------|-------------------|-------------|
| **RX платы** (приём с автопилота) | **GPIO 14** | TX порта телеметрии автопилота |
| **TX платы** (передача на автопилот) | **GPIO 13** | RX порта телеметрии автопилота |
| **GND** | **GND** | GND автопилота |

Важно: **TX автопилота → RX платы (14)**, **RX автопилота → TX платы (13)**. Перепутанные TX/RX — частая причина «нет связи».

### 2.2 Какой порт на автопилоте

Обычно это **TELEM1**, **TELEM2** или **UART** с настроенным MAVLink (например, на Pixhawk — один из разъёмов с подписью TELEM). В параметрах автопилота для этого порта должна быть включена передача MAVLink и задана скорость (см. ниже).

### 2.3 Скорость UART (бодрейт)

В прошивке скорость задаётся в `config.h`:

```c
#define UART_BAUD 57600   // или 115200
```

Значение **должно совпадать** со скоростью порта телеметрии на автопилоте. Часто по умолчанию стоит **57600** (ArduPilot/Pixhawk). Если в Mission Planner нет данных — проверьте в параметрах автопилота (например, `SERIAL1_BAUD`, `SERIAL2_BAUD` в зависимости от порта) и поставьте ту же цифру в `UART_BAUD`.

### 2.4 Уровни напряжения

Логика ESP32 — **3.3 В**. Если автопилот выдаёт 5 В по UART, нужен преобразователь уровней (например, модуль 5V↔3.3V по TX/RX). Многие современные контроллеры уже имеют 3.3 V на телеметрии — тогда можно подключать напрямую.

## 3. Настройка Wi‑Fi (config.h)

### 3.1 Режим точки доступа (MODE_AP) — удобно для полевой работы

В `config.h` должно быть:

```c
#define MODE_AP
#define SSID     "ESP32S3_NEW"
#define PASSWD   "12345678"
#define STATIC_IP IPAddress(192, 168, 2, 1)
#define NETMASK   IPAddress(255, 255, 255, 0)
```

- Плата создаёт сеть **ESP32S3_NEW** с паролем **12345678**.
- IP платы в этой сети: **192.168.2.1**.
- Ноутбук/ПК подключаете к этой Wi‑Fi сети; дальше в Mission Planner указываете адрес **192.168.2.1**.

### 3.2 Режим станции (MODE_STA) — плата подключается к роутеру

Если хотите, чтобы плата подключалась к вашему роутеру:

```c
#define MODE_STA
#define SSID     "Имя_вашей_WiFi"
#define PASSWD   "Пароль_роутера"
#define HOSTNAME "esps3"
```

Тогда IP платы выдаёт роутер (DHCP). Узнать IP можно в Serial Monitor при старте или по имени **esps3** в сети (mDNS), либо в веб-интерфейсе роутера (список устройств).

### 3.3 Порты для MAVLink

В `config.h` уже заданы:

| Параметр | Значение | Назначение |
|----------|----------|------------|
| `SERIAL_TCP_PORT` | 8880 | Порт TCP, к которому подключается Mission Planner |
| `SERIAL_UDP_PORT` | 14550 | Порт UDP (стандарт для GCS MAVLink) |

Менять их нужно только если у вас особые требования (например, второй мост в той же сети).

## 4. Подключение Mission Planner по TCP (рекомендуется)

### 4.1 Шаги по порядку

1. **Проводка**  
   Автопилот подключён к ESP32 по UART (TX→14, RX→13, GND→GND), питание подано.

2. **Включение платы**  
   Дождитесь, пока поднимется Wi‑Fi (в режиме AP — сеть «ESP32S3_NEW»).

3. **Подключение ПК к Wi‑Fi платы**  
   Подключите ноутбук/ПК к сети **ESP32S3_NEW** (пароль из `config.h`, по умолчанию 12345678).

4. **Mission Planner**  
   - Вверху справа выберите тип подключения: **Connection type** → **TCP** (или **TCP Listener** в зависимости от версии).  
   - В поле для адреса введите: **192.168.2.1** (в режиме AP) или IP платы в вашей сети (в режиме STA).  
   - Порт: **8880**.  
   - Нажмите **Connect**.

5. **Проверка**  
   Через несколько секунд в Mission Planner должны появиться:  
   - индикатор подключения (зелёный/жёлтый),  
   - телеметрия (Flight Data),  
   - при необходимости — параметры (Config / Full Parameter List) после запроса.

### 4.2 Если в Mission Planner «No connection» или нет данных

- Убедитесь, что ПК действительно в сети **ESP32S3_NEW** (или что плата в STA и вы в той же сети).  
- Проверьте, что в поле подключения указаны именно **192.168.2.1** и порт **8880** (в режиме AP).  
- Проводка UART: TX автопилота → GPIO 14, RX автопилота → GPIO 13.  
- Совпадение скорости: `UART_BAUD` в `config.h` = скорость порта телеметрии на автопилоте.  
- Откройте Serial Monitor платы (115200 бод): при успешном TCP-подключении должно появиться что-то вроде `TCP client 192.168.2.xxx`.  
- Подождите 3–5 секунд после нажатия Connect — автопилот может начать слать поток после первого HEARTBEAT от GCS.

## 5. Подключение Mission Planner по UDP

### 5.1 Шаги

1. Плата и автопилот подключены, Wi‑Fi платы включён (как в разделе 4).  
2. ПК подключён к той же Wi‑Fi сети, что и плата (AP или общий роутер в STA).  
3. В Mission Planner: **Connection type** → **UDP**.  
4. Порт: **14550** (стандарт для MAVLink GCS).  
5. Нажмите **Connect**.

Прошивка при получении первого UDP-пакета запоминает IP и порт отправителя и дальше шлёт туда все данные с UART (ответ автопилота). Поэтому достаточно нажать Connect — Mission Planner сам отправит пакеты, плата «узнает» адрес и начнёт отдавать телеметрию.

### 5.2 Если по UDP нет связи

- Убедитесь, что ПК и плата в одной подсети.  
- Порт **14550** не должен быть занят другим приложением.  
- В режиме AP IP платы 192.168.2.1; Mission Planner по UDP часто подключается «в никуда» (порт 14550), и ответы приходят с платы на тот же порт ПК — важно, чтобы файрвол не блокировал входящие UDP на 14550.

## 6. Что такое MAVLink и какая информация идёт через мост

MAVLink — протокол обмена сообщениями между наземной станцией (GCS, например Mission Planner) и автопилотом. По сути, мост просто пересылает **байты**: с UART в Wi‑Fi и с Wi‑Fi в UART. Разбор пакетов и отображение делают Mission Planner и автопилот.

### 6.1 Основные типы сообщений

| Сообщение | Направление | Назначение |
|-----------|-------------|------------|
| **HEARTBEAT** | Автопилот → GCS | Подтверждение «жив», тип аппарата, режим. По нему мост определяет «подключение к автопилоту». |
| **PARAM_VALUE** | Автопилот → GCS | Значение одного параметра (в т.ч. SERVO1_REVERS, SERVO3_TRIM, SERVO4_TRIM). |
| **PARAM_REQUEST_READ** | GCS → Автопилот | Запрос значения параметра по имени. |
| **PARAM_SET** | GCS → Автопилот | Установка параметра (имя + значение). |
| Другие (ATTITUDE, GPS_RAW_INT, SYS_STATUS и т.д.) | В обе стороны | Телеметрия, статус, команды. |

Прошивка моста **дополнительно** разбирает входящий с UART поток: при получении HEARTBEAT отмечает «подключён», при PARAM_VALUE — сохраняет значения SERVO1_REVERS, SERVO3_TRIM, SERVO4_TRIM для веб-интерфейса. Всё остальное просто транслируется в Mission Planner и обратно.

### 6.2 Как Mission Planner получает информацию

1. После нажатия **Connect** Mission Planner отправляет свои пакеты (в т.ч. HEARTBEAT от GCS).  
2. Они доходят до платы по Wi‑Fi, плата пересылает их на автопилот по UART.  
3. Автопилот отвечает телеметрией и параметрами по UART на плату.  
4. Плата пересылает эти байты по Wi‑Fi обратно в Mission Planner.  
5. Mission Planner расшифровывает MAVLink и показывает Flight Data, параметры, карту и т.д.

То есть «получение информации через MAVLink» в Mission Planner — это обычное подключение к мосту (TCP или UDP); дальше MP сам запрашивает и обрабатывает нужные сообщения.

## 7. Веб-интерфейс платы (проверка связи и параметры)

Плата поднимает встроенный веб-сервер на порту 80. Удобно использовать для проверки, что мост и автопилот «видят» друг друга, и для работы с параметрами SERVO.

### 7.1 Как открыть

- В режиме AP: в браузере введите **http://192.168.2.1**.  
- В режиме STA: подставьте IP платы в вашей сети (тот же, что для Mission Planner).

### 7.2 Главная страница (/)

Краткие ссылки:

- **Статус (подключение, параметры, лог)** — `/api/status` (JSON).  
- **Параметры SERVO** — страница с формой для SERVO1_REVERS, SERVO3_TRIM, SERVO4_TRIM.  
- **Лог пакетов** — `/api/log` (последние события MAVLink в JSON).

### 7.3 Страница «Параметры SERVO» (/params)

- Показывает: **подключение к автопилоту** (по HEARTBEAT), счётчики **RX/TX** пакетов.  
- Текущие значения **SERVO1_REVERS**, **SERVO3_TRIM**, **SERVO4_TRIM** и признак, получены ли они с автопилота.  
- Кнопка **«Запросить с автопилота»** — отправляет с платы на автопилот запросы PARAM_REQUEST_READ по этим трём параметрам; автопилот пришлёт PARAM_VALUE, значения обновятся на странице.  
- Форма **«Установить»** — отправляет PARAM_SET для выбранных полей (значения уходят на автопилот по UART).

Так вы и **получаете** информацию (запрос параметров и отображение в вебе), и **отправляете** её (установка параметров), при этом Mission Planner тоже может работать с теми же параметрами через своё подключение по TCP/UDP.

### 7.4 API для скриптов или отладки

| URL | Метод | Описание |
|-----|--------|----------|
| `/api/status` | GET | JSON: connected, last_heartbeat_ms, packets_rx/tx, значения и *_known для SERVO1_REVERS, SERVO3_TRIM, SERVO4_TRIM, массив log. |
| `/api/params` | GET | JSON только параметров SERVO и флагов *_known. |
| `/api/params` | POST | Установка: аргументы SERVO1_REVERS, SERVO3_TRIM, SERVO4_TRIM (отправляются PARAM_SET в UART). |
| `/api/param_request` | GET | Отправка трёх PARAM_REQUEST_READ на автопилот. |
| `/api/log` | GET | JSON-массив последних записей лога (события RX/TX, DISCONNECT и т.д.). |

По логу можно убедиться, что идут RX HEARTBEAT, RX PARAM_VALUE и т.д. — значит, MAVLink с автопилотом работает.

## 8. Типичные проблемы и решения

| Проблема | Что проверить |
|----------|----------------|
| Mission Planner не подключается (TCP) | ПК в сети ESP32S3_NEW (или в одной сети с платой в STA). IP 192.168.2.1, порт 8880. Файрвол не блокирует исходящие на 8880. |
| Подключился, но нет телеметрии | Проводка UART (TX автопилота → GPIO 14, RX → GPIO 13, GND). Совпадение UART_BAUD и скорости порта на автопилоте. Включён ли MAVLink на выбранном порту автопилота. |
| В вебе «Подключение: Нет» | С автопилота не приходит HEARTBEAT по UART. Проверьте тот же UART и скорость. После исправления подождите несколько секунд. |
| Параметры на странице /params не обновляются | Нажать «Запросить с автопилота». Убедиться, что Mission Planner или другой GCS не «забивает» канал частыми запросами. Проверить по /api/log, есть ли RX PARAM_VALUE. |
| По UDP ничего не приходит | ПК и плата в одной сети, порт 14550. После Connect подождать 2–3 секунды. Проверить, что в прошивке включён PROTOCOL_UDP. |

## 9. Краткий чек-лист перед первым запуском

1. В **config.h**: нужный режим (MODE_AP или MODE_STA), SSID/пароль, **UART_BAUD** = скорость порта телеметрии автопилота.  
2. **Проводка**: TX автопилота → GPIO 14, RX автопилота → GPIO 13, GND общий.  
3. **Прошивка** залита на ESP32-S3-Box (PlatformIO: Build + Upload).  
4. Плата включена, ПК подключён к её Wi‑Fi (или к одному роутеру с платой в STA).  
5. Mission Planner: тип подключения TCP, адрес 192.168.2.1 (или IP платы в STA), порт 8880 → Connect.  
6. При необходимости: веб **http://192.168.2.1** → «Параметры SERVO» → «Запросить с автопилота» для проверки приёма параметров по MAVLink.

Если все пункты соблюдены, Mission Planner получает информацию через MAVLink по Wi‑Fi-мосту и корректно отображает телеметрию и параметры.

## 10. Примеры кода: получение и отправка MAVLink

Ниже — готовые фрагменты кода для вашей прошивки: как **получать** (парсить) сообщения MAVLink из потока UART и как **отправлять** сообщения на автопилот. Подразумевается, что в проекте подключены `#include <MAVLink.h>` (библиотека okalachev/MAVLink) и порт к автопилоту — `SerialUART` (как в `main.cpp`).

### 10.1 Подключение заголовка и канал

В начале файла (например, `main.cpp`):

```cpp
#include <MAVLink.h>   // библиотека okalachev/MAVLink (common dialect)

// Канал MAVLINK_COMM_0 — один последовательный поток (наш UART).
// Парсер хранит состояние по этому каналу между вызовами.
```

Парсинг идёт по одному байту за раз: вы передаёте каждый принятый байт в `mavlink_parse_char()`. Когда пакет собран целиком, функция вернёт ненулевое значение и заполнит структуру сообщения.

---

### 10.2 Получение информации: парсинг входящего потока

Данные приходят с автопилота по UART. Вы читаете байты (например, в блоке «UART → сеть») и по одному передаёте их в парсер. Когда пакет полностью принят — обрабатываете его по `msg.msgid`.

**Пример: разбор каждого принятого байта с UART**

```cpp
// Буфер уже заполнен из SerialUART: bufFromUART[0..lenFromUART-1]
uint16_t lenFromUART = 0;  // реальная длина после чтения
// ... чтение: while (SerialUART.available()) bufFromUART[lenFromUART++] = SerialUART.read();

mavlink_message_t msg;
mavlink_status_t status;

for (uint16_t i = 0; i < lenFromUART; i++) {
    uint8_t byte = bufFromUART[i];
    if (mavlink_parse_char(MAVLINK_COMM_0, byte, &msg, &status)) {
        // Получен полный пакет MAVLink
        switch (msg.msgid) {
            case MAVLINK_MSG_ID_HEARTBEAT: {
                // Источник: msg.sysid, msg.compid
                // Полезные данные — через decode (если нужно):
                // mavlink_heartbeat_t hb;
                // mavlink_msg_heartbeat_decode(&msg, &hb);
                // hb.custom_mode, hb.base_mode, hb.system_status и т.д.
                mavlinkConnected = true;
                lastHeartbeatMs = millis();
                autopilotSysId = msg.sysid;
                autopilotCompId = msg.compid;
                break;
            }
            case MAVLINK_MSG_ID_PARAM_VALUE: {
                mavlink_param_value_t pv;
                mavlink_msg_param_value_decode(&msg, &pv);
                pv.param_id[15] = '\0';  // строка не более 16 символов
                float value = pv.param_value;
                uint8_t param_type = pv.param_type;
                // Сравнить pv.param_id с "SERVO1_REVERS", "SERVO3_TRIM", "SERVO4_TRIM"
                // и сохранить value в свои переменные
                if (strcmp(pv.param_id, "SERVO1_REVERS") == 0) {
                    paramServo1Revers = value;
                    paramServo1ReversKnown = true;
                } else if (strcmp(pv.param_id, "SERVO3_TRIM") == 0) {
                    paramServo3Trim = value;
                    paramServo3TrimKnown = true;
                } else if (strcmp(pv.param_id, "SERVO4_TRIM") == 0) {
                    paramServo4Trim = value;
                    paramServo4TrimKnown = true;
                }
                break;
            }
            // Другие типы при необходимости:
            // case MAVLINK_MSG_ID_SYS_STATUS: ...
            // case MAVLINK_MSG_ID_GPS_RAW_INT: ...
            default:
                break;
        }
    }
}
```

**Важно:** один и тот же буфер `bufFromUART` после парсинга нужно ещё переслать в сеть (TCP/UDP) — как в текущем `main.cpp`. Парсинг только «заглядывает» в данные, не потребляя их.

---

### 10.3 Отправка информации: формирование и отправка пакетов

Чтобы **отправить** команду или параметр на автопилот, нужно:
1. Собрать сообщение функцией `mavlink_msg_*_pack(...)`.
2. Преобразовать его в байты: `mavlink_msg_to_send_buffer(buf, &msg)`.
3. Записать байты в UART: `SerialUART.write(buf, len)`.

Идентификатор отправителя (GCS): обычно **system_id = 255**, **component_id = 190**. Адресат — автопилот: **target_system** и **target_component** лучше брать из последнего HEARTBEAT (`autopilotSysId`, `autopilotCompId`).

---

#### Пример 1: запрос параметра (PARAM_REQUEST_READ)

Запрашиваем у автопилота значение параметра по имени. Имя — строка до 16 символов, без нуля в конце в самом сообщении (библиотека дополняет нулями при упаковке).

```cpp
#include <string.h>

void requestParameter(const char* param_id) {
    mavlink_message_t msg;
    uint8_t buf[MAVLINK_MAX_PACKET_LEN];

    // -1 = использовать param_id; иначе можно передать индекс параметра
    mavlink_msg_param_request_read_pack(
        255, 190,           // system_id, component_id (GCS)
        &msg,
        autopilotSysId,     // target_system (автопилот)
        autopilotCompId,    // target_component
        param_id,
        -1                  // param_index: -1 = запрос по имени
    );
    uint16_t len = mavlink_msg_to_send_buffer(buf, &msg);
    SerialUART.write(buf, len);
}
// Использование:
// requestParameter("SERVO1_REVERS");
// requestParameter("SERVO3_TRIM");
```

Ответ автопилота придёт сообщением **PARAM_VALUE** — его разбираете в блоке приёма (раздел 10.2).

---

#### Пример 2: установка параметра (PARAM_SET)

Отправка нового значения параметра на автопилот. Тип значения задаётся константой MAVLink (для float — `MAV_PARAM_TYPE_REAL32`).

```cpp
void setParameter(const char* param_id, float value) {
    mavlink_message_t msg;
    uint8_t buf[MAVLINK_MAX_PACKET_LEN];

    mavlink_msg_param_set_pack(
        255, 190,           // GCS
        &msg,
        autopilotSysId,
        autopilotCompId,
        param_id,
        value,
        MAV_PARAM_TYPE_REAL32
    );
    uint16_t len = mavlink_msg_to_send_buffer(buf, &msg);
    SerialUART.write(buf, len);
}
// Использование:
// setParameter("SERVO3_TRIM", 1500.0f);
// setParameter("SERVO1_REVERS", 0.0f);  // 0 или 1 в виде float
```

После успешного приёма автопилот может прислать обновлённый **PARAM_VALUE** — можно обновить свои переменные в парсере.

#### Пример 3: отправка HEARTBEAT (опционально)

Если ваш мост должен сам представляться автопилоту как GCS (например, для стабильного отображения в списке систем), можно периодически слать HEARTBEAT:

```cpp
void sendHeartbeat() {
    mavlink_message_t msg;
    uint8_t buf[MAVLINK_MAX_PACKET_LEN];

    mavlink_msg_heartbeat_pack(
        255, 190,                              // system_id, component_id (GCS)
        &msg,
        MAV_TYPE_GCS,                           // тип: наземная станция
        MAV_AUTOPILOT_INVALID,                  // не автопилот
        MAV_MODE_FLAG_CUSTOM_MODE_ENABLED,      // base_mode
        0,                                      // custom_mode
        MAV_STATE_ACTIVE                        // system_status
    );
    uint16_t len = mavlink_msg_to_send_buffer(buf, &msg);
    SerialUART.write(buf, len);
}
// В loop() вызывать, например, раз в секунду:
// static uint32_t last = 0;
// if (millis() - last >= 1000) { last = millis(); sendHeartbeat(); }
```

Константы `MAV_TYPE_*`, `MAV_STATE_*` и т.д. объявлены в заголовках библиотеки MAVLink (common).

### 10.4 Где в проекте это используется

| Действие | Где в коде |
|---------|------------|
| **Приём и разбор** MAVLink (HEARTBEAT, PARAM_VALUE) | `main.cpp`, блок после чтения из `SerialUART` в `bufFromUART` — цикл с `mavlink_parse_char` и `switch (msg.msgid)`. |
| **Отправка PARAM_REQUEST_READ** | `handleParamRequest()` — запрос SERVO1_REVERS, SERVO3_TRIM, SERVO4_TRIM. |
| **Отправка PARAM_SET** | `handleParamsSet()` — установка значений из веб-формы. |

Добавлять новые типы сообщений (приём или отправку) можно по тому же шаблону: в `switch` по `msg.msgid` вызывать соответствующий `mavlink_msg_*_decode`, а для отправки — `mavlink_msg_*_pack` и `mavlink_msg_to_send_buffer` + `SerialUART.write()`.

## 11. Подсказки, доработки и дополнения

### 11.1 Безопасность

- **Параметры в полёте:** Не меняйте критические параметры (в т.ч. SERVO, калибровки) во время полёта. Сначала проверяйте на стенде с включённым автопилотом и отключёнными моторами.
- **Веб-интерфейс без пароля:** Доступ к страницам и API не защищён. В локальной/домашней сети это обычно допустимо; в общедоступной сети лучше не открывать порт 80 наружу или добавить свою авторизацию (например, проверка пароля в обработчиках).
- **Пароль Wi‑Fi:** В режиме AP смените `PASSWD` в `config.h` на сложный пароль, если сеть доступна посторонним.

### 11.2 Настройки в config.h

| Параметр | Зачем менять |
|----------|----------------|
| `UART_BAUD` | Должен совпадать со скоростью порта телеметрии на автопилоте (часто 57600 или 115200). |
| `BUFFERSIZE` | 1024 достаточно для MAVLink. При частых обрывах или очень плотном потоке можно поднять до 2048. |
| `MAVLINK_LOG_SIZE` | Количество последних записей в логе (по умолчанию 50). Увеличить — если нужна длинная история для отладки. |
| `MAVLINK_HEARTBEAT_TIMEOUT_MS` | Через сколько миллисекунд без HEARTBEAT считать автопилот отключённым (по умолчанию 5000). |

Таймаут HEARTBEAT вынесен в конфиг — см. `config.h` и использование в `main.cpp` (блок «Таймаут MAVLink»).

### 11.3 Имена параметров ArduPilot

- В коде и вебе используются **SERVO1_REVERS** (с одной «R» в REVERS) — так в ArduPilot.
- В других прошивках или старых версиях может быть **SERVO1_REVERSE**. Если параметр не приходит — проверьте точное имя в Mission Planner (Config → Full Parameter List) и при необходимости добавьте вариант в парсер или форму.

### 11.4 Как добавить новые типы сообщений MAVLink

- **Список сообщений и полей:** [MAVLink Common Messages](https://mavlink.io/en/messages/common.html) (и ArduPilot-specific при необходимости).
- **В проекте:** в библиотеке в `.pio/libdeps/esp32s3box/MAVLink/mavlink/common/` есть заголовки `mavlink_msg_*.h` с константами `MAVLINK_MSG_ID_*` и функциями `mavlink_msg_*_decode` / `mavlink_msg_*_pack`.
- Для **приёма:** в цикле парсинга добавить `case MAVLINK_MSG_ID_XXX:` и вызвать соответствующий `mavlink_msg_xxx_decode(&msg, &struct)`.
- Для **отправки:** вызвать `mavlink_msg_xxx_pack(...)`, затем `mavlink_msg_to_send_buffer()` и `SerialUART.write()`.

### 11.5 Дополнительные возможности прошивки

- **OTA (прошивка по Wi‑Fi):** в `config.h` раскомментируйте `#define OTA_HANDLER`. После первой прошивки по USB в PlatformIO можно заливать обновления по сети (см. раздел OTA в курсе или документации PlatformIO).
- **Bluetooth-мост:** раскомментируйте `#define BLUETOOTH` — данные MAVLink пойдут и по Bluetooth (для подключения с телефона и т.п.).
- **Режим STA:** если плата подключается к роутеру (`MODE_STA`), IP можно узнать по mDNS — имя `HOSTNAME` (по умолчанию `esps3`) в сети.

### 11.6 Отладка

- **Serial Monitor:** 115200 бод, COM-порт платы. В логе видны сообщения о TCP-клиентах, Wi‑Fi, ошибках. При `#define DEBUG` вывод идёт в Serial.
- **Веб `/api/log`:** показывает последние события MAVLink (RX HEARTBEAT, RX PARAM_VALUE, TX PARAM_SET и т.д.) — удобно проверять, что пакеты доходят и обрабатываются.
- **Проверка связи:** если в Mission Planner нет телеметрии, а в вебе «Подключение: Да» — значит HEARTBEAT с автопилота доходит до платы; проблема скорее в Wi‑Fi или настройках MP. Если в вебе «Подключение: Нет» — проверяйте UART (провода, скорость, порт на автопилоте).

### 11.7 Рекомендуемый порядок проверки

1. Собрать и прошить плату, подключить автопилот по UART (TX→14, RX→13, GND).  
2. Включить питание, подключить ПК к Wi‑Fi платы (AP) или убедиться, что плата в сети (STA).  
3. Открыть **http://192.168.2.1** (или IP платы) → «Параметры SERVO» → «Запросить с автопилота». Если значения появляются и «Подключение: Да» — UART и MAVLink работают.  
4. В Mission Planner выбрать TCP (или UDP), указать IP и порт, нажать Connect. Должна пойти телеметрия и параметры.  
5. При необходимости менять параметры через веб или через Mission Planner и снова проверять на стенде перед полётом.
